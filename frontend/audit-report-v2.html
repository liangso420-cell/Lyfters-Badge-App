Errores por Severidad

Ordenados por impacto
🔴 CríticoXSS en sistema de toasts

js/utils.js:395 — La función toast() hace el.innerHTML = msg sin escapar el contenido. Cualquier string con HTML llegado desde el servidor (mensajes de error con input del usuario, nombre de badge, etc.) puede ejecutar JavaScript en el navegador del victim.

Fix (1 línea): Cambiar el.innerHTML = msg por el.textContent = msg. Si el toast necesita íconos, construirlos con createElement() en vez de HTML raw. Verificar también cualquier otro innerHTML que reciba strings del servidor sin pasar por U.esc().
🔴 CríticoJWT Secret con fallback débil hardcodeado

backend/app.py:33 — os.getenv("JWT_SECRET", "dev_secret_cambia_esto"). Si el env var no se carga en Render (fallo de deploy o rollback), cualquiera puede forjar tokens JWT con rol god_admin. Este es el bypass total de autenticación. A pesar de que generateValue:true está en render.yaml, el fallback nunca debería existir en producción.

Fix (1 línea): Cambiar a os.getenv("JWT_SECRET") sin default. Si la variable no existe, Flask-JWT lanzará error en el arranque — comportamiento correcto que evita deployments con secret vacío.
🟠 AltoMONGO_URI con fallback a localhost en producción

backend/db.py:31 — os.getenv("MONGO_URI", "mongodb://localhost:27017"). Si la variable no se carga en Render, el backend arranca conectado a una MongoDB que no existe, sin retornar error visible. Las operaciones fallan silenciosamente o escriben en localhost vacío.

Fix: Quitar el default. Agregar validación explícita: mongo_uri = os.getenv("MONGO_URI"); assert mongo_uri, "MONGO_URI requerida". El deploy falla ruidosamente si falta, que es el comportamiento correcto.
🟠 AltoBug en _ROLE_RANK — 'admin' rankeado en 0

backend/routes/workspaces.py:268 — El dict _ROLE_RANK = {"participant": 1, "superadmin": 3, "god_admin": 4} no incluye "admin". El método .get("admin", 0) retorna 0, lo que posiciona a los admins por debajo de los participantes en las comparaciones de rango. Un admin podría no poder gestionar participantes dependiendo del flujo exacto.

Fix (1 carácter): Agregar "admin": 2 al dict. Jerarquía completa: participant:1, admin:2, superadmin:3, god_admin:4.
🟠 AltoCORS demasiado amplio — acepta cualquier *.vercel.app

backend/app.py:46 — La regex r"^https://[a-zA-Z0-9][a-zA-Z0-9\-]*\.vercel\.app$" acepta cualquier subdominio de Vercel. Cualquier desarrollador que deploya a Vercel puede enviar requests autenticadas a la API de Lyfter si tiene un token válido.

Fix: Reemplazar el patrón dinámico por el dominio exacto del frontend: "https://lyfters-badge-app.vercel.app". Agregar también el dominio de GitHub Pages. Mantener la lista explícita, no el wildcard.
🟡 MedioRate limiting en memoria — se pierde al reiniciar el servidor

backend/routes/auth.py:44 — _login_attempts = defaultdict(list) es un dict en memoria del proceso Python. Cada restart de Gunicorn en Render (que ocurre en cada deploy y regularmente por timeout) resetea todos los contadores de intentos fallidos. El rate limit de login no es persistente.

Nota: El rate limiting de flask-limiter usa Redis en producción (configurado en render.yaml), pero este dict adicional en auth.py no. Es redundante con flask-limiter y puede eliminarse si flask-limiter está correctamente configurado con Redis.
🟡 MedioAvatar URLs renderizadas sin validar origen

js/utils.js:545 — Las URLs de avatar (que pueden provenir de Google OAuth o ser subidas por el usuario) se renderizan directamente en <img src="..."> sin validar que el origen sea de confianza. Si una URL comprometida apuntara a un recurso externo, podría filtrar el token de sesión via el header Referer.

Fix recomendado: Validar que las avatarUrl sean data: URIs (base64) o provengan de dominios en whitelist (googleusercontent.com, lh3.googleusercontent.com). Reemplazar URLs que no pasen validación por el avatar de iniciales por defecto.
🟡 MedioFiltro de privacidad duplicado en múltiples endpoints

El mismo objeto {"$or": [{"privacy.show_in_leaderboard": {"$ne": False}}, {"privacy": {"$exists": False}}]} aparece en events.py en tres funciones distintas. Si se agrega un campo de privacidad nuevo, hay que actualizar tres lugares.

Fix: Extraer a constante en events.py: _PRIVACY_VISIBLE = {"$or": [...]} y referenciarla en las tres funciones. Una línea de cambio en vez de tres.
✅ Solucionado en v2Race condition en redención de badges

El viejo find_one + insert_one fue reemplazado por update_one($setOnInsert, upsert=True) con DuplicateKeyError como barrera secundaria. El índice único (user_id, badge_id) en MongoDB garantiza atomicidad incluso bajo requests simultáneas.

Estado: Cerrado. redeem.py:59 usa la versión atómica.
✅ Solucionado en v2Reset password token en respuesta JSON

El endpoint /auth/forgot-password ya no incluye reset_link en la respuesta. Devuelve solo {ok: true}. El email con el link se envía server-side vía Resend.

Estado: Cerrado. auth.py:415.

El puntaje sube de 76 a 78/100, pero la revisión más exhaustiva descubrió dos hallazgos nuevos que son serios: el XSS en el sistema de toasts (utils.js:395) y el bug en _ROLE_RANK donde 'admin' rankea en 0. Ambos son fixes de una línea — pero el XSS en toasts es lo que un evaluador técnico nota en los primeros 10 minutos de revisar el código frontend.

El JWT secret y MONGO_URI sin default son mejoras críticas de producción real pero no son visibles en una demo.

El núcleo del sistema es sólido: multi-tenancy, XP atómico, OAuth verificado server-side, rate limiting real, QR funcional. La arquitectura es correcta para un proyecto de esta escala. Los issues pendientes son configuración y edge cases de seguridad — no diseño defectuoso.
